PROJECT CONTEXT: EDGE-BASED POLYFORM ENGINE (MONOMINO, WITH HOLES)

This project implements polyomino generation using EDGE GEOMETRY, not cell occupancy.

IMPORTANT: Do NOT revert to cell-based representations. This code is explicitly designed
to keep the geometry in boundary cycles so it can extend to dominoes and later tilings.

------------------------------------------------------------

CURRENT STATUS

- Monomino generator is COMPLETE in hole mode and matches OEIS A000105
- Verified correct through at least n = 11 here, and through n = 12 in user testing
- Dump hole counts verified:
  - n = 7 -> 1
  - n = 8 -> 6
- Implementation uses:
  - ordered boundary cycles
  - edge cancellation (antiparallel)
  - multi-cycle shapes (outer + holes)
  - hash-based deduplication
  - canonicalization under rotation + reflection

------------------------------------------------------------

CORE REPRESENTATION

Shape = Poly

Poly:
- one outer boundary cycle
- zero or more hole cycles

Cycle:
- ordered list of vertices (lattice points)
- edges are implicitly unit-length between consecutive vertices
- no duplicate vertices are allowed inside a valid scanned cycle

Edge soup representation is NOT used.

------------------------------------------------------------

KEY INVARIANTS

1. All edges are unit length
2. Cycles are ordered (not unordered edge sets)
3. A valid Poly is a set of simple cycles
4. No floating point arithmetic (integer lattice only)
5. All cycles must be transformed together during canonicalization
6. Outer cycle = maximum absolute signed area
7. Canonical form = [outer] + sorted(holes)
8. Outer orientation = CCW
9. Hole orientation = CW

------------------------------------------------------------

ATTACHMENT MODEL

Tile (currently monomino) is defined by a boundary cycle.

Attachment procedure:
1. Select base edge and tile edge (must be antiparallel)
2. Align tile onto base
3. Detect coincident edges:
   - antiparallel -> cancel
   - parallel -> reject
4. Walk remaining edges into cycles
5. If a duplicate vertex appears while scanning a cycle,
   retry the ambiguous turn with the opposite choice
6. Keep all resulting cycles
7. Canonicalize Poly
8. Deduplicate in hash table

------------------------------------------------------------

CRITICAL TOPOLOGY LESSON

At ambiguous 4-valent turning vertices, a fixed global turn rule is not enough.
The correct practical discriminator in this version is:

-> no duplicate vertices inside a scanned cycle
-> if duplicate occurs, retry with the opposite turn choice

This allows the engine to keep hole-producing configurations correctly.

------------------------------------------------------------

CANONICALIZATION

Do this exactly once at insertion into the next level:

    attach -> extract cycles -> build Poly -> canonicalize Poly -> hash/store

Canonicalization procedure:
1. Apply one symmetry to ALL cycles together
2. Normalize translation once for the whole Poly
3. Identify outer cycle by maximum absolute signed area
4. Force orientations:
   - outer = CCW
   - holes = CW
5. Canonicalize cyclic shift within each cycle
6. Keep outer first
7. Sort only the holes lexicographically
8. Choose the lexicographically minimal transformed Poly

Important bug fix from V2.1.1:
- Do NOT sort all cycles together lexicographically
- That can move a hole before the outer cycle in dumps and canonical form
- Correct rule is:

    Poly = [outer] + sorted(holes)

------------------------------------------------------------

FRONTIER RULE

Frontier scanning must include edges from ALL cycles:
- outer boundary
- hole boundaries

Without this, A000105 descendants are missed.

------------------------------------------------------------

DUMP FORMAT

mono_dump prints:

    <0|1> cycle0 | cycle1 | ...

where:
- 0 = no holes
- 1 = has holes

The outer cycle must print first.

------------------------------------------------------------

WHAT NOT TO DO

- Do NOT introduce floating point math
- Do NOT revert to cell occupancy
- Do NOT transform cycles independently during canonicalization
- Do NOT sort all cycles together and lose outer-first ordering
- Do NOT canonicalize repeatedly at multiple stages
