PROJECT CONTEXT: EDGE-BASED POLYOMINO GENERATION (MONOMINO, NO HOLES)

This project implements a polyomino generator using EDGE GEOMETRY, not cell occupancy.

IMPORTANT: Do NOT revert to cell-based representations. This code is explicitly designed to avoid that.

------------------------------------------------------------

CURRENT STATUS

- Monomino generator is COMPLETE and matches OEIS A000104 (polyominoes WITHOUT holes)
- Verified correct through at least n = 12
- Implementation uses:
  - ordered boundary cycles
  - edge cancellation (antiparallel)
  - hash-based deduplication
  - canonicalization under rotation + reflection

------------------------------------------------------------

CORE REPRESENTATION

Shape = one outer boundary cycle (no holes in current version)

Cycle:
- ordered list of vertices (lattice points)
- edges are implicitly unit-length between consecutive vertices
- orientation must be consistent (outer boundary CCW)

Edge soup representation is NOT used (it caused ambiguity at multi-edge vertices)

------------------------------------------------------------

KEY INVARIANTS

1. All edges are unit length
2. Cycles are ordered (not unordered edge sets)
3. Boundary is a single cycle (no holes yet)
4. No floating point arithmetic (integer lattice only)
5. Canonicalization includes reflections → must reverse cycle direction after reflection

------------------------------------------------------------

ATTACHMENT MODEL

Tile (monomino) is defined by a boundary cycle.

Attachment procedure:
1. Select base edge and tile edge (must be antiparallel)
2. Align tile onto base
3. Detect coincident edges:
   - antiparallel → cancel
   - parallel → reject
4. Validate local vertex structure
5. Build resulting boundary
6. Resolve ambiguous vertices using pairing rule

------------------------------------------------------------

CRITICAL BUG FIX (MOST IMPORTANT LESSON)

At vertices where more than 2 edges meet (corner-touching cases):

- Multiple valid pairings exist
- Wrong pairing causes overcount (A000105 instead of A000104)

Correct rule:

→ Choose pairing that MAXIMIZES cycle count
→ If resulting cycle count > 1, REJECT (no-hole mode)

This is how hole-producing configurations are excluded WITHOUT explicit hole detection.

This rule is essential to match A000104.

------------------------------------------------------------

VALIDATION RULES

- Parallel coincident edges → reject
- Antiparallel coincident edges → cancel
- Vertex-local pairing must be consistent (no crossings)
- Result must form exactly ONE cycle (no holes)

------------------------------------------------------------

CANONICALIZATION

- Apply all 8 symmetries (rotations + reflections)
- After reflection, REVERSE vertex order to restore orientation
- Normalize translation to origin
- Choose lexicographically minimal representation

------------------------------------------------------------

PERFORMANCE STRATEGY

- Frontier scanning (only attach along boundary)
- Hash table for deduplication
- No brute force spatial search
- No global reconstruction of topology

------------------------------------------------------------

WHAT NOT TO DO

- Do NOT introduce floating point math
- Do NOT reconstruct cycles from unordered edges
- Do NOT assume simple polygons (boundary may revisit vertices)
- Do NOT add hole logic yet (next phase will handle that cleanly)
